## 어댑터 기능 구현 방법
- AccountUserDetails, AccountUserDetailsService 와 같은 변환 클래스 새로 생성하여 구현 (어댑터 패턴 적용)
  - 코드를 새로 생성하여 복잡도는 늘어나지만 단일 책임 원칙을 지킬 수 있음
- adaptee에 해당하는 Account, AccountService 클래스에 기존 코드를 수정하여 구현 (어댑터 패턴 적용은 아님)
  - 코드를 새로 생성하지 않아도 되어 복잡도는 줄고 실용적이지만 단일 책임 원칙을 위반

## 어댑터 패턴의 장점과 단점
- 장점
  - 기존 코드를 변경하지 않고 원하는 인터페이스 구현체를 만들어 재사용할 수 있다.
  - 기존 코드가 하던 일과 특정 인터페이스 구현체로 변환하는 작업을 각기 다른 클래스로 분리하여 관리할 수 있다.
- 단점
  - 새 클래스가 생겨 복잡도가 증가할 수 있다. 경우에 따라서는 기존 코드가 해당 인터페이스를 구현하도록 수정하는 것이 좋은 선택이 될 수도 있다.
